## 一、产品定位

### 1.1 核心体验

> 「你不知道谁是队友 — 每一张牌都是试探」

明暗斗地主是 5 人在线棋牌，核心差异化在于**隐藏身份 + 暗号机制**：地主通过暗号牌秘密告知搭档，其余玩家在出牌过程中推理身份、判断阵营。

单局时长：约 5–10 分钟。

### 1.2 目标用户

| 用户群 | 特征 | 需求 |
|--------|------|------|
| 核心用户 | 熟悉斗地主，喜欢社交推理 | 快节奏、有深度、身份机制新鲜感 |
| 泛用户 | 微信棋牌轻度玩家 | 能和朋友随时开一局，门槛低 |
| Demo 阶段 | 内部测试 / 小范围体验 | 能跑完完整流程，体验核心玩法 |

### 1.3 P0 / P1 边界

| 阶段 | 目标 | 时间 |
|------|------|------|
| **P0（当前）** | Demo 可玩：5人能打完一局，关键功能跑通 | 当前 |
| **P1** | 正式上线：真实登录、积分体系、排行榜、签到 | P0 验收后 |

## 二、玩家旅程（Player Journey）

### 2.1 首次体验流程

```
打开 App
  └── LaunchScene（2–5秒加载）
        └── Stub 登录（自动完成，无感知）
              └── HallScene（主大厅）
                    ├── 看到自己的昵称/积分/段位
                    ├── 点击「快速匹配」
                    │     └── 等待室（倒计时 + AI 补位）
                    │           └── 游戏桌
                    │                 ├── 发牌
                    │                 ├── 暗号牌选择（若为地主）
                    │                 ├── 加倍阶段
                    │                 ├── 出牌阶段（核心玩法）
                    │                 └── 结算界面
                    │                       ├── 「再来一局」
                    │                       └── 「返回大厅」
                    └── 点击「好友房」
                          └── 好友房等待室（分享房间码）
                                └── 同上游戏流程
```

### 2.2 各阶段体验目标

| 阶段 | 体验目标 | 关键设计原则 |
|------|---------|------------|
| 启动 | 品牌印象建立，< 5s 进入大厅 | 加载进度可见，不黑屏 |
| 大厅 | 30秒内理解能做什么 | 主操作突出，次要功能不干扰 |
| 匹配 | 等待感不焦虑 | 进度可见 + AI补位兜底 |
| 游戏 | 每次出牌决策有意义 | 信息清晰，操作反馈即时 |
| 结算 | 身份揭晓有惊喜感 | 揭晓动画 + 积分变化清晰 |
| 再来 | 留住用户，不流失 | 「再来一局」CTA 突出 |

## 三、账号与登录系统

### 3.1 P0 — Stub 登录

**机制**: 进入 App 后自动分配临时账号（UUID），无需任何操作。

| 字段 | P0 实现 |
|------|---------|
| 用户 ID | 服务端随机 UUID |
| 昵称 | 随机词组：「神秘玩家 + 4位数字」 |
| 头像 | 默认占位头像（8种颜色随机一种）|
| 积分 | 初始 1000 分 |
| 段位 | 青铜（基于初始积分）|

**Token**: JWT，有效期 24 小时，客户端缓存（localStorage / wx.setStorageSync）。

**P0 不做**: 真实微信授权、绑定手机号、账号注销。

### 3.2 P1 — 微信登录

- `wx.login()` → `code` → 服务端换 `openid` → 创建/更新玩家记录
- 获取微信头像和昵称（需用户授权）
- 账号持久化，积分跨设备同步

## 五、大厅功能

### 5.1 主操作 — 快速匹配

**功能逻辑**:
1. 点击按钮 → 弹出 MatchView（快速匹配等待室）
2. 服务端将玩家加入匹配队列（同段位优先）
3. 等待 30 秒后若不足5人，AI 自动补位
4. 满5人 → 自动跳转 GameScene

**按钮状态**:
- 正常: 可点击，绿色
- 匹配中: 进入 MatchView 后按钮变灰（防重入）
- 网络断开: 按钮禁用 + Toast 提示「网络已断开」

### 5.2 主操作 — 好友房

**功能逻辑**:
1. 点击按钮 → 弹出 MatchView（好友房等待室）
2. 服务端创建房间，生成6位房间码
3. 房主可分享房间码，好友通过码加入
4. 最少 2 人时房主可手动「开始游戏」（其余 AI 补位）
5. 最多 5 人，超出提示「房间已满」

**房间码分享**:
- 点击「分享给好友」→ 调用 `wx.shareAppMessage()`（微信小程序）
- H5 降级: 复制房间码到剪贴板 + Toast「已复制」
- 分享内容: 「我在玩明暗斗地主，来和我一起？房间码：XXXXXX」

**房主权限**:
- 唯一可点「开始游戏」按钮
- 房主离开 → 房间关闭（P0 不做房主转让）

### 5.3 功能宫格

#### 5.3a 游戏规则（P0 实现）

静态页面，展示简化规则说明：
- 游戏人数、牌型说明、暗号机制简介
- 内容来源: `docs/GAME-RULES.md` 摘要版
- 实现: WebView 加载本地 HTML 或 ScrollView + Label

#### 5.3b 排行榜（P1）

P0: 灰色占位，点击 Toast「功能即将上线」  
P1:
- 积分榜：全服 Top 50，按总积分排名
- 本周榜：每周一重置
- 好友榜：仅显示好友（需微信好友关系链）

#### 5.3c 每日签到（P1）

P0: 灰色占位  
P1:
- 日历式签到，连续签到有加成
- 奖励: 积分（50/100/200/300/500/500/1000，连续7天）
- 补签: 不支持

#### 5.3d 限时活动（P1+）

P0: 灰色占位，不实现

## 六、匹配系统

### 6.1 快速匹配逻辑

| 步骤 | 说明 |
|------|------|
| 入队 | 玩家进入等待室，加入匹配队列 |
| 段位分桶 | 优先同段位匹配，±1 段位容忍 |
| 扩展 | 等待 > 15s 开放跨段匹配 |
| AI 补位 | 等待 30s 后不足5人，AI 自动填充空位 |
| 开始条件 | 5人（含AI）全部就位，服务端自动开局 |
| 取消匹配 | 等待中可取消，退出队列 |

**等待室显示**:
- 实时更新「X/5 人已加入」
- 倒计时显示「XX 秒后 AI 补位」
- AI 接入时显示「AI 补位中…」
- 满员显示「即将开始…」后自动跳转

### 6.2 好友房逻辑

| 步骤 | 说明 |
|------|------|
| 创建 | 房主点击「好友房」→ 服务端创建房间，返回房间码 |
| 加入 | 好友输入房间码 → 服务端验证 → 加入房间 |
| 人数限制 | 最多5人，超出返回 error 2003 |
| 开始条件 | 房主点击「开始游戏」（最少2人）|
| AI 补位 | 不足5人时 AI 填充（开始后立即填充）|
| 房主离开 | 房间关闭，所有成员返回大厅 |

### 6.3 断线重连

| 场景 | 行为 |
|------|------|
| 网络短暂中断（< 30s）| 自动重连，恢复游戏状态 |
| 网络长期中断（> 30s）| 提示「断线中，请检查网络」，AI 托管出牌 |
| 重连成功 | 同步最新游戏状态，恢复手牌 |
| 重连失败 | Toast「重连失败，返回大厅」，跳转大厅 |

## 七、游戏内功能

### 7.1 游戏桌信息展示

| 信息 | 位置 | 更新时机 |
|------|------|---------|
| 5个席位（头像/昵称/手牌数）| 各席位节点 | 每次出牌/pass 后 |
| 当前出牌玩家（计时圆环）| 席位头像外圈 | turn_change 消息 |
| 上家出牌展示 | PlayZone 中央 | Schema delta |
| 本人手牌 | HandCardView 底部 | your_hand + 出牌后 |
| 剩余牌数 | 顶部栏（可选）| 实时计算 |
| 倒计时 | 当前出牌席位圆环 | 每秒刷新 |

### 7.2 出牌交互

**出牌流程**:
1. 本人回合 → 手牌可点击
2. 点选牌 → 牌型实时识别，顶部显示牌型名
3. 点「提示」→ 服务端返回推荐牌，蓝色高亮
4. 点「出牌」→ 发送 play_cards，等待服务端响应
5. 成功 → 牌从手牌飞出到 PlayZone
6. 失败 → Toast 错误提示，手牌保持选中

**不要（pass）**:
- 自由轮（isNewRound=true）：「不要」按钮禁用
- 跟牌轮：「不要」按钮可用
- 点击 → 发送 pass，当前席位出现「不要」气泡

**超时托管**:
- 30s 倒计时归零 → 服务端自动出牌（AI 托管一次）
- 连续 3 次超时 → 全程 AI 托管，席位显示「AI」灰标
- 中途任意出牌 → 解除托管

### 7.3 暗号牌选择（仅地主）

- 进入 `landlord_select` 阶段自动弹出
- 展示所有合法暗号牌（3-10点，4个花色）
- 玩家选择一张确认 → 服务端记录
- 30s 内未选 → 服务端自动随机选一张
- 其他玩家不可见选择结果（只有搭档出该牌时才揭露）

### 7.4 加倍阶段

**流程**:
1. 地主先选（×1 或 ×2）
2. 地主选完后，其余4人同时选择
3. 所有人选完 → 显示汇总结果（1.5s）→ 自动进入出牌阶段
4. 超时（15s）→ 自动视为 ×1

**显示规则**:
- 地主选完前：非地主按钮禁用，显示「等待地主选择…」
- 地主选完后：显示地主选择结果，非地主按钮激活
- 结果汇总：「席位X: ×1」逐行显示

### 7.5 结算界面

**身份揭晓**:
- 所有玩家身份同时揭晓（地主/搭档/平民）
- 揭晓动画：全屏短暂遮罩 + 「身份揭晓」大字 + 席位标签出现
- 搭档在揭晓前对其他平民玩家也是未知的

**积分展示**:
- 每个玩家的积分变化（+/- 金色/红色）
- 倍率明细：底分 × 地主倍 × 玩家加倍倍数 × 炸弹系数 × 春天系数

**再来一局**:
- 30s 窗口期
- 好友房：显示「X/Y 人同意」，全部同意则重开（不够人 AI 补）
- 快速匹配：点击后重新排队，不等其他人

## 八、网络异常与错误处理

### 8.1 错误码对应提示

| 错误码 | 含义 | 用户提示 |
|--------|------|---------|
| 1001 | 牌型不合法 | Toast「牌型不合法，请重新选择」|
| 1002 | 压不过上家 | Toast「压不过上家，请选更大的牌或不要」|
| 1003 | 不是你的回合 | 静默（正常防御，不提示）|
| 1004 | 手里没有这张牌 | Toast「出牌异常，请刷新」|
| 2001 | 房间不存在 | Toast「房间不存在或已结束」|
| 2002 | 房间已满 | Toast「房间已满（5/5）」|
| 2003 | 无权限操作 | Toast「只有房主可以执行此操作」|

### 8.2 网络状态提示

| 状态 | 视觉 | 行为 |
|------|------|------|
| 连接正常 | 无提示 | 正常游戏 |
| 断线中 | 顶部红色横幅「网络已断开，尝试重连…」| AI 托管出牌 |
| 重连中 | 黄色横幅 + 旋转图标 | 等待重连 |
| 重连成功 | 绿色横幅 2s 后消失 | 恢复游戏状态 |
| 重连失败 | Toast「重连失败」| 跳转大厅 |

### 8.3 异常场景处理

| 场景 | 处理方式 |
|------|---------|
| 服务端重启 | 客户端检测连接断开，走重连流程 |
| 5人全部离线 | 房间自动关闭，数据不写库 |
| 结算失败 | Toast「结算异常，本局不计分」，返回大厅 |
| 加入不存在的房间码 | Toast「房间码无效」，停留等待室 |
